# R15 Branding and Asset Contract

OpenPortalKit customer workspaces use `apps/web/src/lib/branding.json` as the controlled public identity contract.
The manifest is versioned as `opk.branding.v1` and is generated by `opk new`. It covers the site identity, approved
assets, color tokens, typography preset, primary navigation, and footer links without allowing runtime CSS or
JavaScript injection.

## Manifest

```json
{
  "schemaVersion": "opk.branding.v1",
  "site": {
    "name": "Atlas Public Portal",
    "shortName": "APP",
    "description": "Atlas publishes accessible, traceable public content and structured data.",
    "locale": "en"
  },
  "assets": {
    "logo": null,
    "favicon": { "src": "/favicon.ico", "width": 256, "height": 256 },
    "socialImage": {
      "src": "/examples/corporate.webp",
      "alt": "Engineers inspect an electric motor in a bright manufacturing laboratory.",
      "width": 1920,
      "height": 1080
    }
  },
  "colors": {
    "accent": "#087c78",
    "accentStrong": "#075f5c",
    "surface": "#ffffff",
    "surfaceMuted": "#f4f7f6",
    "text": "#151918"
  },
  "typography": { "preset": "editorial" },
  "navigation": [{ "label": "Publications", "href": "#publications" }],
  "footer": { "copyright": "Atlas Public Portal", "links": [] }
}
```

Typography presets are `editorial`, `modern`, and `institutional`. They are intentionally bounded choices; teams
that need a custom type system should add licensed font files under `apps/web/public/` and review source-level CSS.

Links may be fragments, root-relative paths, or absolute HTTPS URLs without embedded credentials. The Web runtime
uses the manifest in generated customer workspaces and live mode. The framework repository keeps its five demo
profiles and their distinct layouts unchanged for visual evaluation.

## Asset Rules

- Put logos and social images under `apps/web/public/`; their manifest paths begin with `/`.
- `/favicon.ico` maps to Next.js convention file `apps/web/src/app/favicon.ico`.
- Logos allow SVG, PNG, WebP, and JPEG up to 1 MiB.
- Favicons allow ICO, PNG, and SVG up to 256 KiB.
- Social images allow PNG, WebP, and JPEG up to 2 MiB.
- Declared dimensions must match the file. Dimensions are limited to 4096 by 4096 pixels.
- Logo and social-image alternative text is required. When `logo` is `null`, the header renders `site.shortName` as
  an accessible visual fallback and retains the full site name as the home-link label.
- SVG parsing prohibits DTDs, scripts, `foreignObject`, event handlers, and external references.
- Asset paths cannot contain traversal, backslashes, query strings, fragments, or protocol-relative URLs.

The JSON Schema is `schemas/branding-manifest.v1.schema.json`. Run the semantic and physical-file validator after
every identity or asset change:

```bash
./tools/opk branding validate --root .
./tools/opk branding validate --root . --format json
```

Validation also enforces WCAG-oriented contrast for normal text, muted surfaces, strong accents, and white text on
the main accent. It fails closed before packaging when an asset is absent, oversized, malformed, unsafe, or has
dimensions that disagree with the manifest.

## Source Customization Boundary

The manifest is not a design canvas, DAM, upload API, or low-code configuration system. Responsive composition,
custom components, licensed fonts, and deeper visual changes remain normal reviewed React/CSS source work. Never add
arbitrary script, HTML, or CSS fields to the manifest.

Branding changes alter public HTML metadata, Open Graph images, navigation, and visual identity, so they must be
reviewed and tested from an approved customer-workspace commit. They do not mutate CMS records and therefore do not
create an application audit event. Publishing audit/outbox/revalidation behavior remains unchanged. Dashboard
metrics and structured-data provenance are unaffected; AgentSEO/public-origin acceptance must still be rerun because
navigation and metadata are public outputs.

## Verification

Run `opk branding validate`, Web lint/build, and desktop/mobile Playwright tests. Confirm logo and text fallback,
favicon, social preview metadata, keyboard navigation, long site names, mobile header fit, and every external link.
The release checklist treats the branding manifest and approved customization commit as artifact provenance.
